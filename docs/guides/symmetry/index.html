Symmetry Savings

Reduce FLOP costs when your tensors have symmetry.

You will learn:

  • How to declare full and non-full tensor symmetries with we.as_symmetric()
  • How to generate example tensors for arbitrary permutation groups
  • When to use we.random.symmetric() (sample + project) versus we.symmetrize() (project existing data)
  • How slicing, reductions, and binary pointwise ops preserve, weaken, or drop symmetry metadata
  • When to re-tag results with we.as_symmetric() after conservative propagation

Why symmetry matters

Many tensors contain repeated structure. A symmetric matrix has only n * (n + 1) / 2 unique elements instead of n^2, and higher-order tensors with permutation symmetry can shrink the effective element count even more. When whest knows a tensor's symmetry, it charges FLOPs based on unique elements instead of dense ones.

OperationDense costSymmetry-aware costWhy it drops
we.exp(s2_matrix)n^2n * (n + 1) / 2only unique matrix entries matter
we.einsum('ki,kj->ij', x, x, symmetric_axes=[(0, 1)])m * n^2m * n * (n + 1) / 2the repeated x operand lets whest detect symmetric output and reduce the cost
we.einsum('i,j,k->ijk', v, v, v)n^3symmetry-reducedrepeated operands induce output symmetry

By contrast, symmetric_axes on an einsum output tags the result for downstream operations; it does not reduce that einsum's own cost by itself.

Quick start

we.as_symmetric() validates the data first. After that, whest propagates symmetry metadata algebraically through many operations. Unary pointwise ops preserve symmetry-aware costs and keep the same symmetric axes. For non-full groups such as C_k or D_k, the current implementation widens the metadata to full symmetry on those axes. Slicing, reductions, and binary pointwise ops can weaken it or remove it entirely.

How to declare symmetry

Full symmetric shorthand with symmetric_axes

Use symmetric_axes when the tensor is invariant under every permutation of a set of axes.

This is the most common declaration: full S_2 symmetry on matrix axes (0, 1).

Multiple independent full symmetric groups

This declares one full symmetric group on axes (0, 1) and another on (2, 3).

Explicit full symmetric groups with PermutationGroup.symmetric

This is equivalent in meaning to symmetric_axes=(0, 1, 2), but explicit groups are useful once you want to inspect or combine them directly.

Cyclic symmetry with PermutationGroup.cyclic

C_3 means rotations are allowed, but reflections are not. This is weaker than S_3, so it usually gives fewer savings.

Dihedral symmetry with PermutationGroup.dihedral

D_4 includes both rotations and reflections of a four-position structure.

Arbitrary subgroups from custom generators

Use this form when the built-in constructors do not describe your symmetry.

Multiple explicit groups on one tensor

This is the explicit-group equivalent of symmetric_axes=[(0, 1), (2, 3)].

Generating example data with the Reynolds operator

When you want example tensors for arbitrary groups, prefer we.random.symmetric. It is a handy helper that samples from a distribution and applies the Reynolds operator. Use we.symmetrize when you already have concrete data to symmetrize, and we.as_symmetric when you already have concrete data to validate and tag.

R_G(T) = (1 / |G|) * sum_{g in G} g · T

This helper is ideal for docs, tests, and experiments:

  • it works for S_k, C_k, D_k, and custom generator sets
  • prefer we.random.symmetric() for synthetic data generation
  • we.random.symmetric internally samples data and calls we.symmetrize so the projection and validation behavior is identical.
  • approximate costs (meaningful estimate):
    • we.random.symmetric: C_dist(n) + |G| * n + n + validation
    • we.symmetrize: |G| * n + n + validation
      with n total elements and |G| group order.
  • in exact arithmetic it projects onto the invariant subspace, and in practice we.as_symmetric() validates the result with its usual validation tolerances
  • it keeps examples consistent across symmetry classes

The propagation examples below assume import whest as we; generated tensors above use we.random.symmetric(...) and the explicit transform helper is we.symmetrize(...).

Symmetry propagation at a glance

Symmetry propagation is conservative. Whest keeps symmetry metadata only when the operation's structure guarantees that the output still respects the surviving group.

OperationResult typeRule
we.exp(s3_tensor)SymmetricTensorunary pointwise ops preserve symmetric axes; non-full groups are currently widened to full symmetry on those axes
we.add(s3_tensor, c3_tensor)SymmetricTensor or ndarraybinary pointwise ops keep the intersection of both operands' groups
s2_matrix * 3SymmetricTensorscalar binary ops preserve the tensor's groups
s2_matrix[0]ndarrayinteger indexing removes one axis; no nontrivial group survives
s2_matrix[:3, :3]SymmetricTensorequal-size slices can preserve symmetry
s2_matrix[:3, :2]ndarrayunequal-size slices break symmetry between those axes
s2_matrix[we.array([0, 1])]ndarrayadvanced indexing drops symmetry conservatively
we.sum(s3_tensor, axis=0)SymmetricTensorreductions keep the surviving setwise stabilizer subgroup
we.sum(d4_tensor, axis=(1, 3))SymmetricTensorreductions can keep a proper subgroup like C_2
s2_matrix @ s2_matrixndarraymatrix products are not assumed symmetric in general

we.exp(c3_tensor) still gets symmetry-aware cost savings, but the result is currently tagged as full symmetry on (0, 1, 2) rather than preserving the original C_3 subgroup exactly.

Slicing rules

Slicing uses the pointwise stabilizer of the removed axes. Informally: every removed axis must stay fixed under any surviving group element.

What happens here:

  • same_size_slice stays symmetric because both surviving axes still have the same size
  • different_size_slice loses symmetry because the two axes no longer match
  • expanded_s2_matrix keeps the same S_2 action, but the axes are renumbered from (0, 1) to (1, 2)
  • advanced_index_slice returns a plain ndarray; whest does not attempt to propagate symmetry through array/list indexing

Ellipsis behaves like NumPy's normal expansion rules. It can change which axes remain, but it does not change the propagation rule itself.

The difference between full and non-full groups matters immediately:

  • s3_slice keeps an S_2 subgroup on the surviving axes
  • c3_slice loses all nontrivial symmetry, because C_3 has no non-identity element that fixes one point

Reduction rules

Reductions use the setwise stabilizer of the reduced axes. Informally: reduced axes are allowed to permute among themselves, because summation treats all positions along the reduced set equivalently.

What happens here:

  • s3_reduced keeps an S_2 subgroup on the remaining axes
  • c3_reduced loses all nontrivial symmetry
  • c4_reduced keeps a C_2 subgroup
  • c4_keepdims keeps the same surviving subgroup, but the output axes stay in their original tensor positions because keepdims=True

Reducing an axis that is not in a symmetry group leaves that group alone, apart from any axis renumbering caused by the removed dimension.

Binary pointwise ops and broadcasting

Binary pointwise ops keep only the symmetry present in both operands. For general groups, that means element-set intersection on matching output axes, not just matching tuples of axis numbers.

intersection_tensor keeps C_3, because C_3 is the common subgroup of S_3 and C_3.

Multiple groups are handled independently:

shared_group_tensor keeps symmetry on (0, 1) and loses the independent group on (2, 3).

Broadcasting matters too:

Before group intersection, any axis stretched from size 1 to a larger output size is removed from the candidate group. That is why broadcasted singleton axes do not automatically stay symmetric after a binary op.

Warnings, conservative behavior, and re-tagging

Whest propagates symmetry metadata conservatively. When the operation does not guarantee that the declared symmetry survives, the result falls back to a plain ndarray.

row_slice is a plain ndarray, not a SymmetricTensor. That is expected.

SymmetryLossWarning tells you that metadata was dropped or weakened during an operation. If you know more about the result than the conservative propagation rule does, you can re-tag it with we.as_symmetric().

One important caveat: the current implementation does not report every possible partial weakening via SymmetryLossWarning. Treat the warning system as helpful guidance, not as a complete audit of every same-axis subgroup change.

Under the hood

The propagation rules are easier to predict if you keep three ideas in mind:

  • Unary pointwise ops currently rebuild results from symmetric_axes, so they preserve the same symmetric axes but not exact subgroup identity for C_k, D_k, or custom groups
  • Slicing uses the pointwise stabilizer of the removed axes
  • Reductions use the setwise stabilizer of the reduced axes
  • Binary pointwise ops use intersection of both operands' groups after broadcast alignment

After computing the surviving subgroup, whest restricts it to the axes still present in the output and remaps those axes to the output tensor's numbering.

That is why:

  • slicing one axis of S_3 leaves S_2
  • slicing one axis of C_3 leaves nothing nontrivial
  • reducing {1, 3} of C_4 can still leave C_2
