# Client-Server Sync: Design Spec

**Date:** 2026-04-13
**Status:** Approved
**Goal:** Bring mechestim-client and mechestim-server back in sync with the core library, and ensure they can never silently drift again.

## Problem

The core library has evolved (stats distributions, permutation groups, symmetric tensors, new error types, FLOP weight changes) while the client and server packages have not kept pace. The client's registry is 8 days stale (450 entries vs 482+ in core), error constructors have incompatible signatures, entire submodules are missing, and the test suites can't even run due to dependency resolution issues.

### Specific Divergences Found

| Area | Core | Client | Server |
|------|------|--------|--------|
| Registry entries | 482+ | 450 (stale snapshot) | N/A (imports core directly) |
| Stats submodule | 8 distributions, 24 methods | Missing entirely | Dispatch works via getattr but untested |
| Permutation classes | Permutation, PermutationGroup, Cycle | Missing | No serialization support |
| SymmetricTensor | ndarray subclass with SymmetryInfo | Missing | No metadata in pack_result |
| `SymmetryError` constructor | `axes`, `atol`, `rtol` params | `dims` param (incompatible) | N/A |
| `UnsupportedFunctionError` | Present | Missing | Not in error handler chain |
| `SymmetryLossWarning` | Present | Missing | N/A |
| OpRecord fields | `op_name`, `subscripts`, `shapes`, `flop_cost`, `cumulative`, `namespace` | Missing: `subscripts`, `shapes`, `namespace` | N/A |
| `budget_reset()` | Exported | Missing | N/A |
| linalg proxies | 36+ functions | Only `svd` explicit | Works via getattr |
| Server dep resolution | N/A | N/A | Can't resolve `mechestim>=0.2.0` (not on PyPI) |

## Solution: Hybrid (Generation + Contract Tests)

### Part 1: Code Generation — `scripts/sync_client.py`

A single script that reads the live core library and generates derived client artifacts.

**Modes:**
- `uv run scripts/sync_client.py` — regenerate files in-place
- `uv run scripts/sync_client.py --check` — exit non-zero if anything would change (CI gate)

**Generated files:**

| Target file | Source of truth | Content |
|-------------|----------------|---------|
| `mechestim-client/src/mechestim/_registry_data.py` | `mechestim._registry.REGISTRY` | `FUNCTION_CATEGORIES` dict: op name → category |
| `mechestim-client/src/mechestim/errors.py` | `mechestim.errors` module | All error/warning classes with matching constructors, `raise_from_response()`, `_ERROR_MAP` |
| `mechestim-client/src/mechestim/stats/__init__.py` | `mechestim.stats.__all__` | Proxy distribution objects with `.pdf()`, `.cdf()`, `.ppf()` methods |
| `mechestim-client/src/mechestim/linalg/__init__.py` | Registry entries with `linalg.` prefix | Auto-generated proxies for all linalg ops |
| `mechestim-client/src/mechestim/_perm_group.py` | `mechestim._perm_group` module source | Copy of pure-Python permutation group classes (Permutation, PermutationGroup, Cycle) |

**What stays hand-written (NOT generated):**
- `_connection.py` — ZMQ socket management
- `_protocol.py` — msgpack encoding/decoding
- `_remote_array.py` — RemoteArray/RemoteScalar proxy classes
- `_budget.py` — client-side budget tracking (but OpRecord fields will be verified by contract tests)
- `_comms_tracker.py` — communication metrics
- `_display.py` — budget visualization
- `_getattr.py` — module __getattr__ factory
- `_math_compat.py` / `_constants.py` — math constants
- `__init__.py` — top-level module (but will be updated to import stats)

**Generation approach:**
- Script imports `mechestim` (the core library from `src/`)
- Introspects `REGISTRY`, `errors` module, `stats.__all__`
- Writes Python source files with `# AUTO-GENERATED by scripts/sync_client.py — DO NOT EDIT` header
- Uses deterministic output (sorted keys, stable formatting) so `--check` mode is reliable

### Part 2: Serialization for New Types

#### Permutation/Cycle/PermutationGroup (client → server)

These classes are **pure Python with no numpy dependency**. Strategy: the sync script generates a copy of `_perm_group.py` into the client package (with auto-generated header), sourced from the core's version. This keeps it in the generation pipeline so drift is caught. Participants construct these locally; they're serialized when passed as arguments to operations like `einsum(symmetry=...)`.

**Wire format (client → server):**

```python
# Permutation([2, 0, 1]) →
{"__permutation__": [2, 0, 1]}

# PermutationGroup with generators →
{"__perm_group__": {"generators": [[1, 0, 2], [2, 1, 0]], "degree": 3, "axes": [0, 1, 2]}}

# Cycle(0, 2)(1, 3) → convert to Permutation first, then serialize
{"__permutation__": [2, 3, 0, 1]}
```

**Client changes:**
- `_encode_arg()` in `_remote_array.py`: detect Permutation/PermutationGroup/Cycle → serialize to dict
- `__init__.py`: export Permutation, PermutationGroup, Cycle

**Server changes:**
- `_resolve_arg()` in `_request_handler.py`: detect `__permutation__`/`__perm_group__` dicts → reconstruct real objects

#### SymmetricTensor / SymmetryInfo (server → client)

SymmetricTensor subclasses `np.ndarray`. The array data goes through the normal handle mechanism; symmetry metadata is included alongside.

**Wire format (server → client):**

```python
# SymmetricTensor result →
{
    "status": "ok",
    "result": {
        "id": "a5",
        "shape": [3, 3],
        "dtype": "float64",
        "symmetry_info": {
            "symmetric_axes": [[0, 1]],
            "shape": [3, 3]
        }
    },
    "budget": {...}
}
```

**Server changes:**
- `_pack_result()` in `_request_handler.py`: `isinstance(result, me.SymmetricTensor)` check before generic ndarray check; include symmetry metadata

**Client changes:**
- `_result_from_response()` in `_remote_array.py`: when `symmetry_info` key is present, store it as a property on the RemoteArray
- `RemoteArray`: add optional `symmetry_info` property (None by default)
- Include `SymmetryInfo` dataclass in client (pure Python, no numpy)

### Part 3: Contract Tests

Three test categories with different speed/scope trade-offs:

#### 3a. API Surface Parity Tests (fast, no server)

Location: `tests/test_client_server_parity.py` (runs from root project)

Tests:
- Every entry in core's `REGISTRY` exists in client's `FUNCTION_CATEGORIES`
- Entry counts match exactly (no stale threshold like `>= 482`)
- Every error class in core's `errors.py` exists in client's `errors.py`
- Error constructor signatures are compatible (same parameter names)
- Every submodule in core (linalg, fft, random, stats) has a matching client submodule
- OpRecord fields match between core and client
- Permutation/Cycle/PermutationGroup public methods exist in client's copy
- `budget_reset()` exists in client
- Client `__init__.py` imports the `stats` submodule

#### 3b. Serialization Round-trip Tests (fast, no server)

Location: `tests/test_serialization_parity.py`

Tests:
- Permutation → `_encode_arg()` → `_resolve_arg()` → same array form
- PermutationGroup → encode → decode → same generators, degree, axes
- Cycle → encode → decode → same permutation
- SymmetryInfo → server pack → client unpack → same symmetric_axes
- All existing types (ndarray handle, scalar, tuple-of-arrays) still round-trip correctly

#### 3c. End-to-End Integration Tests (slow, server subprocess)

Location: `mechestim-client/tests/test_full_integration.py`

Tests:
- One representative operation per category: pointwise (`add`), reduction (`sum`), linalg (`svd`), random (`normal`), stats (`norm.pdf`), einsum
- Stats distribution round-trip: `me.stats.norm.pdf(array)` returns correct numerical values
- Einsum with `symmetry=PermutationGroup(...)` works end-to-end
- Error propagation: `BudgetExhaustedError` from server arrives at client with correct `op_name`, `flop_cost`, `flops_remaining`
- SymmetricTensor round-trip: operation returns SymmetricTensor on server, client receives RemoteArray with `symmetry_info` populated

### Part 4: Server-side Fixes

#### 4a. Dependency Resolution

Add to `mechestim-server/pyproject.toml`:

```toml
[tool.uv.sources]
mechestim = { path = "../", editable = true }
```

This lets the server resolve the local core library in dev and CI.

#### 4b. `_resolve_arg()` — New Type Deserialization

In `_request_handler.py`, add dict-format detection for:
- `{"__permutation__": [...]}` → `me.Permutation([...])`
- `{"__perm_group__": {...}}` → reconstruct `me.PermutationGroup` from generators

#### 4c. `_pack_result()` — SymmetricTensor Handling

Add `isinstance(result, me.SymmetricTensor)` check **before** the generic `np.ndarray` check (since SymmetricTensor is a subclass). When detected, include `symmetry_info` dict in the response metadata.

#### 4d. Error Handler Chain

Add `UnsupportedFunctionError` to the exception handler in `RequestHandler.handle()`:

```python
except me.UnsupportedFunctionError as e:
    return {"status": "error", "error_type": "UnsupportedFunctionError", "message": str(e)}
```

### Part 5: CI Integration

**New jobs in `.github/workflows/ci.yml`:**

```yaml
client-server-sync:
  # Fast gate — pure import checks, no subprocess
  steps:
    - Install core library
    - Run: python scripts/sync_client.py --check
    - Run: pytest tests/test_client_server_parity.py tests/test_serialization_parity.py

client-server-integration:
  needs: [client-server-sync]
  # Slow — requires server subprocess
  steps:
    - Install core, client (from mechestim-client/), server (from mechestim-server/)
    - Run: pytest mechestim-client/tests/test_full_integration.py
```

**Makefile targets:**
- `make check-sync` — runs generation check + parity tests
- `make test-integration` — runs full end-to-end suite

### Part 6: Client `__init__.py` Updates

Beyond the generated files, the client's `__init__.py` needs manual updates:
- Import `stats` submodule alongside `linalg`, `fft`, `random`
- Export `Permutation`, `PermutationGroup`, `Cycle` from `_perm_group`
- Export `SymmetryInfo` from a new `_symmetric_info.py` module — a pure-Python version of the dataclass (no numpy imports, no `SymmetricTensor`). Contains only the `SymmetryInfo` dataclass with `symmetric_axes` and `shape` fields, without the `unique_elements`/`symmetry_factor` computed properties (those need PermutationGroup.burnside_unique_count which operates on real tensors). The client's SymmetryInfo is a **read-only metadata container** — it records what symmetry the server reported, not a computation engine.
- Export `UnsupportedFunctionError`, `SymmetryLossWarning` from errors
- Export `budget_reset` from `_budget`

## Out of Scope

- Publishing mechestim to PyPI (server dependency solved via local path)
- Testing submodule on client (not needed for sandbox)
- Client-side `BudgetAccumulator` (server-only feature)
- `as_symmetric()` / `is_symmetric()` on client (these operate on real arrays)

## Success Criteria

1. `scripts/sync_client.py --check` passes (no generation drift)
2. All API surface parity tests pass
3. All serialization round-trip tests pass
4. All end-to-end integration tests pass (every operation category)
5. Server tests pass with local core dependency
6. Client tests pass (existing + new)
7. CI jobs gate PRs on all of the above
